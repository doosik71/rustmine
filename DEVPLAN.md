# 개발 계획서 (DEVPLAN)

## 1. 개요

본 문서는 `TRD.md`와 `ARCHITECTURE.md`를 기반으로 RustMine의 모듈별 개발 일정과 단계별 목표를 정의한다. 일정은 **MVP → Alpha → Beta → 1.0**의 4단계로 구성하며, 총 24주(약 6개월) 로드맵을 기준으로 한다.

## 2. 단계별 목표

### 2.1 MVP (1~8주)

- 오버월드 단일 바이옴 (완료)
- 기본 블록 배치/파괴 (완료)
- 단일 플레이어 (완료)
- 최소 렌더링 및 입력 처리 (완료)

### 2.2 Alpha (9~14주)

- 멀티플레이 기본 기능
- 엔티티(플레이어/기본 몬스터)
- 기본 제작 시스템

### 2.3 Beta (15~20주)

- 확장 월드 생성 (바이옴 다양화)
- 기본 UI/오디오 적용
- 성능 최적화 1차

### 2.4 1.0 (21~24주)

- 컨텐츠 확장 (광물/도구/동물)
- 안정성 강화 및 테스트 완료
- 최적화 2차

## 2.5 효율적인 구현 순서 (결정)

핵심 원칙은 **데이터 → 시뮬레이션 → 시각화 → 사용자 조작 → 저장/네트워크**의 흐름을 따르는 것이다.  
의존성이 강한 렌더/네트워크를 뒤로 두고, 월드/블록/물리 기반을 먼저 완성하여 리워크를 최소화한다.

1. **Core 기반**: 수학/유틸/직렬화/로깅
2. **World 데이터**: 블록/청크/높이맵/단일 바이옴 지형
3. **Entity/Physics 최소**: 플레이어 캡슐 + 충돌
4. **Client Loop**: 입력/카메라/단순 렌더(디버그 뷰 포함)
5. **World Interact**: 블록 파괴/배치/드롭
6. **UI 최소**: HUD/핫바
7. **Save/Load**: 월드 저장/로드
8. **Network**: 서버 권한 동기화

이 순서는 MVP의 리스크를 가장 빨리 줄이고, 이후 Alpha 기능 추가 시 병목을 줄인다.

## 3. 모듈별 개발 계획

### 3.1 공통(Core)

- 1~2주: 수학/유틸/직렬화 기본 구축 (완료)
- 3~4주: ECS 기반 컴포넌트 구조 확립
- 5~6주: 물리/충돌/경로 탐색 기초

### 3.2 월드(World)

- 1~2주: 청크 구조 설계 및 저장 포맷
- 3~4주: 기본 블록/지형 생성 (완료)
- 5~6주: 바이옴 적용 및 구조물 1종
- 15~18주: 바이옴 확장 및 생성 개선

### 3.3 렌더(Render)

- 1~2주: 렌더링 백엔드 선택(wgpu) (완료)
- 3~4주: 청크 메싱 및 기본 셰이더 (완료)
- 5~8주: 조명/안개/파티클
- 15~20주: 최적화, LOD 적용

### 3.4 엔티티(Entity)

- 5~8주: 플레이어 이동/물리 (완료)
- 9~12주: 기본 몬스터 AI
- 15~20주: 동물/추가 몬스터

### 3.5 네트워크(Net)

- 7~8주: 패킷 구조 설계
- 9~12주: 서버/클라이언트 동기화
- 13~14주: 지연 보정 및 최적화

### 3.6 UI

- 13~16주: HUD 및 인벤토리
- 17~20주: 메뉴/채팅

### 3.7 오디오

- 15~18주: 효과음
- 19~20주: 배경음

### 3.8 서버(Server)

- 7~8주: 서버 런타임 뼈대
- 9~14주: 월드/엔티티 시뮬레이션
- 21~24주: 안정화 및 관리 기능

### 3.9 클라이언트(Client)

- 1~4주: 기본 입력/렌더 루프 (완료)
- 5~8주: 싱글플레이 통합
- 9~14주: 멀티플레이 통합

## 3.10 구현 순서 기반 세부 태스크

### 단계 A: Core/World 기반 (1~2주)

- 수학 벡터/행렬 타입 (완료)
- 로깅/프로파일링 유틸
- 청크/블록 자료구조 확정 (완료)
- 단일 바이옴 높이맵 기반 지형 생성 (완료)

### 단계 B: 최소 엔티티/물리 (3~4주)

- 플레이어 이동/점프/중력 (완료)
- 블록 AABB 충돌
- 카메라/시점 변환 (완료)

### 단계 C: 렌더/클라 루프 (5~6주)

- 청크 메싱 최소 구현 (완료)
- 기본 렌더 파이프라인 (완료)
- 입력 → 이동 → 렌더 통합 루프 (완료)
- 카메라 기준 5x5 청크 렌더링 (완료)
- 비동기 메싱 + 청크 캐시 + 부분 갱신 (완료)

### 단계 D: 월드 상호작용 (7~8주)

- 블록 파괴/배치 (완료)
- 인벤토리 최소(핫바) (완료)
- MVP 완성

### 단계 E: 저장/네트워크 (9~12주)

- 월드 저장/로드
- 서버 런타임/틱 처리
- 클라-서버 동기화

### 단계 F: 콘텐츠/확장 (13~24주)

- 몬스터/동물/제작/바이옴 확장
- UI/오디오/성능 최적화

## 5. 리스크 및 대응

- **렌더링 성능 저하** → 청크 메싱 비동기 처리

- **콘텐츠 부족** → 우선순위 기반 단계적 추가

## 8. Bevy 전환 로드맵 (구체화)

### 8.1 전환 원칙

- **시점**: MVP 완료 직후 ~ Alpha 시작 전
- **범위**: 렌더/입력/엔티티/오디오/자산 파이프라인을 Bevy로 이관
- **보존**: 월드/블록/청크 데이터 구조 및 생성 로직은 최대한 유지

### 8.2 단계별 전환 계획

#### Phase 1: 준비 (MVP 완료 직후, 1주)

- Bevy 버전 선정 및 프로젝트 스캐폴딩
- 기존 모듈 분리 상태 점검(core/world/entity/render)
- Bevy 플러그인 구조 설계(월드, 렌더, 입력, 네트워크)

#### Phase 2: 렌더/입력 이관 (2주)

- 카메라/이동 입력을 Bevy 시스템으로 전환
- 청크 메시 생성 결과를 Bevy Mesh로 변환
- 최소 렌더 파이프라인 확보

#### Phase 3: 엔티티/물리 이관 (2주)

- 플레이어/몬스터 엔티티를 Bevy ECS로 이식
- 충돌 처리(간단 AABB)와 이동 시스템 통합
- 몬스터 AI(상태 머신) 시스템 추가

#### Phase 4: UI/오디오/자산 (1~2주)

- HUD/핫바 UI 이관
- 기본 효과음/배경음 파이프라인 연결
- 에셋 로딩/관리 체계 구축

#### Phase 5: 네트워크/서버 연동 (2주)

- 기존 네트워크 모듈과 Bevy 스케줄 연동
- 서버-클라 상태 동기화 시스템 통합

### 8.3 산출물

- Bevy 기반 클라이언트 프로토타입
- 기존 월드 데이터 재사용 가능한 브릿지 계층
- 전환 전/후 기능 비교 체크리스트
