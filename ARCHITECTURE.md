# 시스템 설계서 (ARCHITECTURE)

## 1. 개요

본 문서는 `TRD.md`의 요구사항을 충족하기 위한 Rust 기반 마인크래프트 유사 게임(RustMine)의 시스템 설계를 정의한다.
목표는 클라이언트/서버/월드/렌더링/네트워크/저장 시스템을 모듈화하고, 단계적 확장이 가능한 아키텍처를 제시하는 것이다.

## 2. 상위 아키텍처

### 2.1 구성 개요

- **클라이언트**: 렌더링, 입력, UI, 오디오, 네트워크 수신
- **서버**: 월드 시뮬레이션, 엔티티 AI, 게임 룰, 저장 관리
- **공통 라이브러리**: 데이터 정의, 수학, 직렬화, 물리, 충돌

### 2.2 논리 블록 다이어그램(텍스트)

- Client
  - Render
  - UI
  - Input
  - Audio
  - Net Client
  - World Cache
- Server
  - World Sim
  - Entity AI
  - Net Server
  - Chunk Manager
  - Save/Load
- Shared
  - Core
  - Data Registry
  - Physics
  - Serialization

## 3. 런타임 흐름

### 3.1 클라이언트 틱

1. 입력 수집
2. 카메라/플레이어 상태 갱신
3. 서버 패킷 처리 및 상태 동기화
4. 렌더링 파이프라인 실행
5. 오디오 업데이트

### 3.2 서버 틱(20 TPS 기준)

1. 네트워크 패킷 수신 및 검증
2. 플레이어/엔티티 물리 업데이트
3. AI/게임 룰 적용
4. 블록 변경 반영 및 이벤트 처리
5. 월드 저장 큐 처리
6. 상태 스냅샷 및 패킷 송신

## 4. 모듈 설계

### 4.1 `core`

- 수학(벡터, 행렬, AABB)
- 유틸(타이머, 로그)
- 직렬화/역직렬화

### 4.2 `world`

- 청크(16x16x384)
- 블록 팔레트/런길이 압축
- 바이옴/구조물
- 월드 생성기

### 4.3 `entity`

- 엔티티 베이스, 컴포넌트
- 물리/충돌
- AI 행동 트리 또는 상태 머신

### 4.4 `render`

- 렌더 파이프라인 (지오메트리, 조명, 파티클)
- 청크 메싱 및 LOD
- 셰이더 관리

### 4.5 `net`

- 패킷 정의 및 버전 관리
- 서버 권한 상태 동기화
- 압축/암호화(필요시)

### 4.6 `audio`

- 사운드 이벤트/믹싱

### 4.7 `ui`

- HUD, 인벤토리, 메뉴

### 4.8 `client`

- 렌더/입력/네트워크 조합 런타임

### 4.9 `server`

- 월드 시뮬레이션 및 네트워크 허브

## 5. 데이터 설계

### 5.1 데이터 레지스트리

- 블록/아이템/레시피는 JSON 기반
- 런타임 로드 후 ID 매핑

### 5.2 저장 포맷

- Region 파일 포맷 유사 구조
- 청크 단위 압축 저장
- 주기적 저장 + 종료 시 플러시

## 6. 네트워크 설계

- **서버 권한 모델**: 모든 월드 변경은 서버가 최종 승인
- **동기화**: 위치/블록/엔티티 상태 패킷 분리
- **지연 허용**: 150ms까지 보정

## 7. 성능 설계

- 청크 메싱은 비동기 큐 처리
- 렌더링과 시뮬레이션 분리
- 월드 I/O는 백그라운드 스레드

## 8. 확장성 고려

- 모드/플러그인 시스템을 위한 데이터 기반 설계
- 레드스톤/포션/마법부여 등 향후 확장 고려

## 9. 보안/안정성

- 패킷 검증 및 리플레이 방지
- 월드 저장 오류 시 복구 가능한 백업 유지
